# 🧾 Commit Conventions (STRICT)

We follow **Conventional Commits (simplified OSS version)**.

## Format

```
<type>: <short description>
```

## Types

- feat → New feature
- fix → Bug fix
- docs → Documentation changes
- style → Formatting / UI updates
- refactor → Code restructuring
- chore → Maintenance
- perf → Performance improvement

## Examples

```
feat: add contribution dashboard section
fix: correct broken image links in README
docs: improve contributing guide
refactor: restructure assets folder
```

---

# 🔁 Pull Request Workflow

## Before Submitting

Ensure:

- Code is clean and readable
- No broken markdown
- Assets optimized
- Proper commit messages used

## Steps

```
git add .
git commit -m "feat: your change description"
git push origin feature/your-change
```

Then open a Pull Request.

---

## PR Title Format

```
[type]: short description
```

Examples:

- feat: improve README layout
- fix: resolve badge alignment
- docs: update CONTRIBUTING.md

---

## PR Description Template

```
## 📌 What does this PR do?

Explain your changes clearly.

## 🎯 Why is this needed?

Motivation behind this change.

## 🧪 Testing

Describe how you tested it.

## 📸 Screenshots

Add visuals if applicable.

## 🔗 Related Issues

Closes #ID
```

# 📏 Code & Content Standards

## Markdown Standards

- Use proper heading hierarchy
- Avoid unnecessary emojis
- Maintain readability
- Keep structure consistent

---

## Asset Standards

- Store all assets in `/assets`
- Prefer:
  - `.svg` → icons/logos
  - `.webp` → images
- Compress before upload

---

## File Naming Convention

Use:

```
lowercase-kebab-case
```

Examples:

```
profile-banner.webp
github-stats.svg
contribution-graph.svg
```
